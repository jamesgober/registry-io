# Release v0.6.0 — Performance Verification
**Date:** 2026-05-19
**Compare:** `v0.5.0...v0.6.0`

## Summary

This release closes the performance verification phase of the roadmap.
The full benchmark suite has been executed against the dev-machine
baseline, every Performance Contract target is met with significant
headroom, and the zero-allocation property of `SyncRegistry::notify` is
now verified by a dhat-backed test rather than asserted by inspection.

Phase **0.6.0** is complete. The next milestone (0.7.0 — hardening,
proptest, fuzz) is the gate to the 1.0 release-candidate work.

## Headline numbers (dev-machine baseline)

| Operation                                           | Target       | Measured       |
|-----------------------------------------------------|-------------:|---------------:|
| Sync `notify`, 1 handler, 1 thread                  |     `<20 ns` |    **10.1 ns** |
| Sync `notify`, 4 handlers, 1 thread                 |     `<50 ns` |    **12.5 ns** |
| Sync `notify`, 16 handlers, 1 thread                |    `<200 ns` |    **26.0 ns** |
| Sync `notify`, 4 handlers, 16 threads contended     |     `<50 ns` |    **24.7 ns** |

register / unregister at `N = 0, 16, 100, 1000`, async at
`N = 0, 1, 4, 16` for both dispatch modes), plus the reproduction
commands.

## Added

- **`benches/contention.rs`** — dedicated thread-contention sweep for
  `SyncRegistry::notify`. Matrix of `{1, 4, 16, 64}` notifying threads ×
  `{1, 4, 16}` handlers. Each benchmark uses `iter_custom` plus
  `thread::scope` so the worker spawn happens inside the timed region;
  reported nanoseconds are *per `notify` call*.
- **`tests/zero_alloc.rs`** — dhat-backed heap accounting test that
  verifies `notify()` allocates **zero** blocks across 100 000 calls,
  exercised in two scenarios (empty registry, 8-handler registry).
  Both scenarios share one `dhat::Profiler` because dhat allows at most
  one profiler per process.
- **`dhat-heap` Cargo feature** — when enabled, the zero-alloc test
  swaps the global allocator to `dhat::Alloc`. Off by default so it
  never affects regular tests or benchmarks. `dhat = "0.3"` lives in
  `[dev-dependencies]` and never propagates to downstream crates.
- **`docs/PERFORMANCE.md`** — extensively rewritten from the prior
  scaffold-shaped document into an authoritative reference:
  - Headline sync table vs Performance Contract targets.
  - Full contention sweep table (1 → 64 threads × 1 → 16 handlers).
  - Async notify table for both concurrent and sequential dispatch.
  - Register / unregister slow-path latency at four scales.
  - Cost model, hot-path code walk-through, slow-path rcu explainer,
    memory footprint accounting.
  - "Choosing a dispatch mode" guidance for the async API.
  - Reproduction commands for every benchmark and the zero-alloc test.

## Changed

### Performance tuning

- **`src/async_registry/mod.rs`** — `AsyncRegistry::notify` (concurrent)
  rewritten to do a single pass over the handler snapshot, building
  parallel `ids` and `wrapped` vectors instead of an intermediate
  `pairs` Vec. Saves one Vec allocation per `notify()` call.
  Functionally identical; tests unaffected.

### Cargo / build

- **`Cargo.toml`** — version `0.5.0` → `0.6.0`. New `[[bench]]` entry
  for `contention`. New `dhat = "0.3"` dev-dependency and
  `dhat-heap = []` feature.

### CI

- **`.github/workflows/ci.yml`** — set
  `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true` in the workflow `env`
  block. Opts `actions/cache@v4` (and any other still-Node-20 JS
  actions) into Node.js 24 today, silencing the "Node.js 20 actions are
  deprecated" warning surfaced on every CI run.
