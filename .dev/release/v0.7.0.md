# Release v0.7.0 — Hardening
**Date:** 2026-05-19
**Compare:** `v0.6.0...v0.7.0`

## Summary

This release closes the hardening phase of the roadmap. Property-based
invariant tests now exercise the registry across random operation
sequences; an `Arc::strong_count` canary verifies that handler closures
are released after every register / unregister cycle; a `cargo-fuzz`
scaffold provides two fuzz targets ready to soak; and the threat model
and verification posture are written down in `docs/SECURITY.md`.

Phase **0.7.0** is complete with one explicit deferral: the 1
CPU-hour fuzz soak listed in the roadmap is parked until the 0.9.0
release-candidate phase, where it joins the rest of the pre-1.0
release gates.

## Added

### Property-based invariant tests

`tests/proptest_invariants.rs` — six properties, each shrunk by
`proptest 1.x`:

- `register_then_unregister_all_leaves_registry_empty(n in 0..=200)` —
  full round-trip is loss-less.
- `notify_fires_exactly_once_per_registered_handler(n in 0..=200)` —
  fire count = handler count, twice in a row.
- `handler_ids_are_unique_across_churn(ops)` — every id issued during
  random register / unregister-most-recent sequences is new.
- `handler_count_matches_bookkeeping_across_random_ops(ops)` —
  observable count tracks the test's parallel `Vec<HandlerId>`.
- `unregister_of_already_removed_id_always_returns_false` — stale ids
  remain rejected after arbitrary subsequent activity.
- `clear_then_register_resumes_with_fresh_unique_ids` — `clear()` does
  not reset the id generator; pre-clear ids never collide with
  post-clear ids.

### Memory-leak verification

`tests/leak_check.rs` — three `Arc::strong_count` canary scenarios:

- 10 000 register / unregister cycles; canary stays `<= 4` (the tiny
  slack accommodates `arc-swap`'s thread-local snapshot cache).
- 100 registrations followed by `clear()`; canary stays `<= 4`.
- Registry drop while 50 handlers are live; canary returns to exactly
  `1`.

### Fuzz scaffold

`fuzz/` — `cargo-fuzz`-ready package:

- `fuzz/Cargo.toml` declares two `[[bin]]` fuzz targets with
  `libfuzzer-sys = "0.4"` and `arbitrary = "1"`.
- `fuzz/fuzz_targets/handler_churn.rs` — drives a random sequence of
  register / unregister / clear / notify / handler_count / contains
  operations against a fresh registry, with bookkeeping assertions
  after each step. Panicky handlers are part of the rotation.
- `fuzz/fuzz_targets/event_payload.rs` — registers a fixed handler set
  (including a tag-conditional panic) and dispatches adversarial event
  payloads chunked out of the fuzzer's input stream.
- `fuzz/.gitignore` excludes `target/`, `corpus/`, `artifacts/`,
  `coverage/`.

The fuzz package is intentionally **not** wired into the root
Cargo workspace, so `cargo build` / `cargo test` from the crate root
ignore it. Invocations:

```bash
rustup install nightly
cargo install cargo-fuzz
cargo +nightly fuzz run handler_churn -- -max_total_time=300
cargo +nightly fuzz run event_payload -- -max_total_time=300
```

### Security documentation

`docs/SECURITY.md` — published threat model:

- Scope: in-process primitive; out of scope: untrusted-input parsing,
  network/IPC boundaries, confidentiality of payloads, AuthN/AuthZ.
- Defended threats with mitigations: panic-tear-down via
  `catch_unwind`, callback-panics via inner `catch_unwind`,
  long-lived leaks via `tests/leak_check.rs`, allocation creep via
  `tests/zero_alloc.rs`, supply-chain via `cargo audit` + `cargo deny`
  (planned for RC phase).
- Unsafe-code discipline: public API uses zero `unsafe`; trusted
  dependencies (`arc-swap`, `std`) audited upstream.
- Panic-isolation walk-through with code excerpts and test pointers.
- Fuzz methodology, run commands, corpus policy, findings status.
- Memory-leak and zero-allocation verification summaries.
- CI gate inventory + vulnerability reporting process.

## Changed

### CI

- **`.github/workflows/ci.yml`** — bumped `actions/cache@v4` to
  `actions/cache@v5`. v5's manifest declares `using: node24`, so the
  `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24` env var added in 0.6.0 is no
  longer required and was removed. The "Node.js 20 actions are
  deprecated" warning that surfaced on every CI run is now gone.

### Cargo

- **`Cargo.toml`** — version `0.6.0` → `0.7.0`; new dev-dependency
  `proptest = { version = "1", default-features = false, features = ["std"] }`.

### Documentation

- **`README.md`** — Status section reordered with v0.7.0 hardening
  bullets first; v0.6.0 perf numbers retained underneath.
- **`.dev/ROADMAP.md`** — Phase 0.7.0 checkboxes flipped, exit criteria
  annotated; the 1-CPU-hour fuzz soak explicitly deferred to phase
  0.9.0.
