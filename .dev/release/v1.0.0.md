# Release v1.0.0 — Stable
**Date:** 2026-05-19
**Compare:** `v0.9.0...v1.0.0`

## Summary

`registry-io 1.0.0` is the **first stable release**. The public API is
frozen for the lifetime of v1.x under the contract in
[`docs/STABILITY-1.0.md`](../../docs/STABILITY-1.0.md). Every
behavioral guarantee, performance number, and panic-isolation rule
that the 0.x line accumulated is now locked in.

**No new public API.** v1.0.0 promotes the surface finalized in
v0.9.0 to stable; upgrading from v0.9.0 → v1.0.0 requires no
compile-time or behavioral changes.

The RC track (`1.0.0-rc.1`) was intentionally skipped. The full 0.x
line — six numbered milestones from 0.4.0 to 0.9.0 — served that
purpose: cumulative hardening, performance verification, and a full
documentation pass.

## Public API frozen at 1.0.0

All items in [`docs/STABILITY-1.0.md`](../../docs/STABILITY-1.0.md)'s
"Public API surface" section are now under the v1.x stability contract:

- `HandlerId`, `PanicInfo<'a>`, `SyncRegistry<E>`, `HandlerGuard<E>`,
  `AsyncRegistry<E>` *(feat: async)*, `AsyncHandlerGuard<E>`
  *(feat: async)*, `pub const VERSION`, `pub mod sync`,
  `pub mod r#async`.
- 15 sync methods + 16 async methods + 4 guard methods, all with
  their signatures and semantics locked.
- `Default + Debug + Send + Sync` impls where applicable, all locked.

Cargo metadata also frozen for v1.x:

- Edition `2024`
- MSRV `1.85` (was tentative; now binding for v1.x)
- License `Apache-2.0 OR MIT`
- Features: `std`, `sync`, `async`, `hybrid`, `metrics` (reserved),
  `dhat-heap` (dev-only). All additive.

## Added in 1.0.0 specifically

There is **no new public API** in this release. The added items are
maintenance/documentation:

- **`docs/STABILITY-1.0.md`** rewritten from "this is what we plan"
  to "this is in effect". Every guarantee clause now reads as a
  present-tense statement of the binding contract.
- **`docs/SECURITY.md`** updated:
  - The deferred 1-CPU-hour fuzz soak (carried from phase 0.7.0) is
    recorded as performed, clean, no findings.
  - `cargo audit`, `cargo deny check`, and `cargo public-api diff`
    moved from "planned for RC phase" to "maintenance gates run
    before each v1.x.y publish" with their v1.0.0 status (all clean).
- **`README.md` Status section** rewritten:
  - "Active development" / "API not yet frozen" / "minor releases
    may break" language **removed**.
  - Now leads with **"Stable. Production-ready."**
  - Surfaces the headline measured numbers and the locked-in
    guarantees inline.
- **`CHANGELOG.md`** `[1.0.0]` block consolidates the full v1.0
  contract for users coming straight to the stable release without
  reading every 0.x changelog block.

## Skipped

A `1.0.0-rc.1` release candidate. The 0.x line covered the same
ground a numbered RC would have:

- **0.4.0** — `SyncRegistry` foundation (lock-free reads,
  priority ordering, panic isolation, RAII guards)
- **0.5.0** — `AsyncRegistry` (concurrent + sequential dispatch,
  panic isolation across `.await`)
- **0.6.0** — performance verification (all Contract targets met
  with measured headroom; dhat-verified zero-alloc)
- **0.7.0** — hardening (proptest invariants, `Arc::strong_count`
  leak canary, `cargo-fuzz` scaffold, published threat model)
- **0.8.0** — integration validation (four pattern examples,
  `docs/PATTERNS.md`)
- **0.9.0** — final documentation pass (stability contract,
  architecture, platform notes, rustdoc audit)

Cutting an RC after that would have re-soaked content the 0.x line
already shook out across published releases.

## Downstream consumers

This crate is the priority blocker for these portfolio crates'
notification primitives — they can now begin consuming
`registry-io = "1.0"`:

- `config-lib` — hot-reload change notifications
- `fsys` — journal observer hooks, write-commit notifications
- `audit-trail` — sink notification fan-out
- `metrics-lib` — exporter notification
- `DISTRO` — transaction state-change hooks (commit, abort, recovery)
- `hive-server` — connection lifecycle events
- `bouncer-io` — rate-limit threshold notifications

The integration patterns each of these would use are documented in
[`docs/PATTERNS.md`](../../docs/PATTERNS.md) with runnable examples.

## Post-1.0 backlog

The following are explicitly **not** v1.0 features but remain on the
post-1.0 roadmap (see `.dev/ROADMAP.md#post-1.0-backlog`):

- `HybridRegistry<E>` — mixed sync + async handlers
- Conditional dispatch (per-handler filters/predicates)
- Weak references (auto-cleanup of dropped subscribers)
- Built-in metrics observability behind a feature flag
- `no_std` support (separate code path with static handler arrays)
- Hierarchical registries (parent/child for event scoping)
- Tracing integration (auto-emit spans for handler dispatch)

Anything in this list ships as v1.1.0+, never breaks the v1.0
contract.

## Out of scope forever

- Cross-process delivery (use NATS, Redis, or another message broker)
- Distributed registries (consensus is a different problem)
- Persistent event log (use an event store)
- Schema management (events are typed Rust values)
- Authentication / authorization (handle-based access is intentional)
