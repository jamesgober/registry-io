# Release v0.8.0 — Integration Validation
**Date:** 2026-05-19
**Compare:** `v0.7.0...v0.8.0`

## Summary

This release closes the integration-validation phase of the roadmap.
The four canonical use cases the crate was built to serve — hot-reload
notification, audit-log fan-out, metric-event collection, and
transaction state-change hooks — are now each documented end-to-end in
`docs/PATTERNS.md` and backed by a runnable example under `examples/`.

This release also includes a **full documentation accuracy sweep**.
Stale version strings (`v0.5.0` footers in `API.md`, `v0.6.0` in
`PERFORMANCE.md`, `v0.4` install snippets, `html_root_url = "0.5.0"`
in `src/lib.rs` that had drifted three releases out of date) were
caught and corrected. The release-checklist has been updated to make
this automatic going forward.

Phase **0.8.0** is complete. The next milestone (0.9.0 — final docs
+ release candidate cut) is the gate to the 1.0 release.

## Added

### Integration-pattern examples (4)

- **`examples/pattern_hot_reload.rs`** — config-lib-style hot reload.
  A `Config` struct owns both `arc_swap::ArcSwap<Snapshot>` and
  `Arc<SyncRegistry<Snapshot>>`. `Config::mutate` swaps the snapshot
  and `notify`s subscribers; subscribers re-derive their local state
  from each new snapshot. Two subscribers in the example: an HTTP
  timeout observer and a feature-flag transition counter (using
  `AtomicBool` for interior mutability inside the `Fn` handler).
- **`examples/pattern_audit_fanout.rs`** — three sinks
  (`StdoutSink`, `FileSink`, `CriticalAlertSink`) each owning a
  `HandlerGuard<AuditEvent>` so their registration lifetimes are tied
  to their owning struct. Demonstrates `register_guard` + RAII
  detach + panic-isolated dispatch.
- **`examples/pattern_metric_event.rs`** — two handlers: a
  zero-allocation atomic counter aggregator (`Counters` struct of
  `AtomicU64`s) and a batching collector (`Mutex<Vec<MetricEvent>>`)
  for an out-of-band exporter to drain. Drives 1 000 simulated
  requests across `RequestStarted`/`RequestCompleted`/`Error`
  variants.
- **`examples/pattern_transaction_hooks.rs`** — priority-ordered
  transaction lifecycle hooks. WAL flush (`priority = 1000`) →
  cache invalidate (`500`) → replication ship (default `0`) →
  metrics record (`-100`). Drives a `Begun → Prepared → Committed`
  sequence plus an `Aborted` sequence and prints the resulting hook
  execution log.

All four examples compile cleanly under `RUSTFLAGS="-D warnings"`.

### Integration-pattern reference

- **`docs/PATTERNS.md`** — four sections, one per pattern. Each
  states the **problem**, sketches the **solution shape** in
  pseudocode, lists the **alternatives** (mpsc per subscriber,
  tokio broadcast, `Arc<Mutex<Vec<_>>>`, polling) with what each
  trades off, and cross-links to the runnable example. Closes with:
  - A **sync-vs-async decision matrix** keyed to handler workload.
  - A **mistakes-to-avoid** list (slow work in sync handlers,
    re-entrant `notify`, oversized closure captures, cross-registry
    id reuse).

### API.md expansion

The previously-condensed async block — "`unregister / clear / contains
/ handler_count / is_empty`" and "`on_panic / clear_panic_callback`" —
is now **seven individual sections** matching the depth of the sync
side. Every public method now has its own description, signature,
parameter list, return-value note, and at least one runnable example
(many have two or three). The TOC was updated accordingly.

### Documentation cross-links

- `README.md`'s **Documentation** section now lists `PATTERNS.md` and
  `SECURITY.md` alongside `API.md` and `PERFORMANCE.md`.
